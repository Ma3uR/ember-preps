# SECTION 1 — Generative AI Fundamentals (Senior Depth)

## 1.1 How LLMs actually work

**Transformers in 60 seconds.** A decoder-only Transformer (the architecture of GPT-5, Claude 4.x, Llama 4, Gemini 3) is a stack of identical blocks; each block has two sublayers — multi-head self-attention and a position-wise feed-forward network — with residual connections and pre-norm (modern LLMs use RMSNorm). Inputs are tokenized, embedded into vectors, augmented with positional information. Modern LLMs use **RoPE (Rotary Positional Embeddings)**, which rotate Q/K vectors in 2D subspaces and allow length extrapolation — this is what makes Claude Opus 4.6 and Sonnet 4.6's 1M-token context window practical. The model is trained on next-token prediction; instruction-tuned chat models add SFT + preference optimisation (DPO/GRPO) on top.

**Attention (the single most-asked question).** For each token, the model projects it into Query, Key and Value vectors. Attention weights are `softmax(QKᵀ/√d_k)·V`. Intuition: each token "queries" every other token's key to decide how much of that token's value to pull in. Multi-head attention runs this in parallel across H subspaces, then concatenates. **Why this matters:** attention is what gives Transformers long-range dependency without recurrence, and it's also why context cost is O(n²) in the naive form — which is why modern LLMs use **grouped-query attention (GQA)** and **KV-caching** to make autoregressive generation O(n) per new token.

**Encoder vs decoder vs encoder-decoder.**
- **Encoder-only** (BERT, sentence-transformers, BGE, Cohere `embed-v3`): bidirectional attention, used for embeddings, classification, retrieval.
- **Decoder-only** (GPT, Claude, Llama, Gemini, DeepSeek, Qwen): causal mask, autoregressive next-token. The dominant chat-model architecture.
- **Encoder-decoder** (T5, original Transformer): used for translation and some SQL models (CodeT5, PICARD). Largely replaced by decoder-only at scale.

**Tokenization.**
- **BPE** (GPT-2/3/4/5): Byte-Pair Encoding, merges most-frequent pairs.
- **WordPiece** (BERT): like BPE but uses likelihood instead of frequency for merges.
- **SentencePiece / Unigram** (T5, Llama, Gemini): trains on raw text, language-agnostic, includes whitespace as a token.
- **Why it matters in interview:** explains why "1234567890" costs ~3 tokens and "the" costs 1, why non-English text costs more, why exact-string prompt matching is brittle.

**Sampling parameters.**
| Param | Effect | When |
|---|---|---|
| `temperature` (0–2) | Divides logits before softmax | 0 for SQL/code/JSON; 0.3–0.7 chat; 0.9+ creative |
| `top_p` (nucleus) | Sample only from smallest set of tokens whose cum-prob ≥ p | 0.9–0.95 most common |
| `top_k` | Sample only from k highest-prob tokens | Less common in modern APIs |
| `frequency/presence_penalty` | Penalise repeats | Avoid combining with low temp |
| `max_tokens` | Hard output cap | Always set |
| `seed` | Deterministic across calls | Use in eval pipelines |

> **Senior tip:** Don't combine `temperature=0` with high `top_p` and `top_k`. Pick one. For text-to-SQL, `temperature=0` plus structured outputs / JSON mode is canonical.

## 1.2 Embeddings

A fixed-length dense vector representation such that semantically similar text has geometrically close vectors. Generated by a forward pass through an encoder trained with contrastive objectives.

**Dimensions (2026):** OpenAI `text-embedding-3-small` (1536-d), `-3-large` (3072-d, Matryoshka-truncatable), Voyage AI `voyage-3` (1024-d), Cohere `embed-v3` (1024-d), open-source BGE-M3 (1024-d), Nomic-embed-v2.

**Similarity metrics.**
- **Cosine** = `(a·b)/(‖a‖‖b‖)`. Range [-1,1]. Magnitude-invariant. **Default for text** — most embedding models L2-normalise outputs so cosine ≡ dot product.
- **Dot product** = `a·b`. Faster (no norm). Equivalent to cosine only when normalised.
- **Euclidean (L2)** = `‖a-b‖`. Image embeddings, some clustering, rarely text RAG.

> **Senior tip:** With normalised vectors on pgvector, use `<#>` (negative inner product) over `<=>` (cosine distance) for speed; same recall.

## 1.3 Vector databases (2026 production landscape)

| Vector DB | Best when | Watch-outs |
|---|---|---|
| **pgvector** | Already on Postgres, ≤10–50M vectors, want ACID + joins | Single-node ceiling; HNSW tuning matters; manual BM25 |
| **Pinecone** | Managed-only, zero-ops, fast prototyping | Recall not tunable; vendor lock-in; cost compounds |
| **Qdrant** | OSS, Rust speed, filtered search | Newer ecosystem |
| **Weaviate** | Native hybrid (BM25+vector), GraphQL schema | More knobs; resource-heavy |
| **Milvus / Zilliz** | Billion-scale, mature sharding | Operational complexity |
| **Chroma** | Local dev, embedded use, small projects | Not for serious prod scale |
| **LanceDB** | Multi-modal, columnar | Newer |

**Decision rule (the answer to give):** "If you're already on Postgres and under ~10M vectors, start with pgvector — HNSW indexes (pgvector 0.5+) are competitive with dedicated stores per Supabase's published benchmarks. Move to Qdrant for filter-heavy workloads, Weaviate if you want native hybrid, Pinecone if you don't want to run infra."

**Q&A pair.** "Why pgvector at scale?" — "Per the Vecstore 2026 production comparison, pgvector HNSW returns results in 5-8ms at our scale; at ~1M vectors, throughput is roughly 640 QPS versus 1620+ for Pinecone/Qdrant/Weaviate per Second Talent's 2026 benchmarks. Below ~10M vectors and where you want transactional consistency with your application data, the ops simplicity wins. Above ~50M you're tuning HNSW aggressively or moving to a dedicated engine."

## 1.4 RAG — Naive → Advanced

**Naive RAG.**
```
Doc → chunk → embed → vector store
Query → embed → top-k → stuff into prompt → LLM → answer
```

**Failure modes:** chunks lose context, lexical recall misses (exact IDs, version numbers), low precision (near-duplicates), bad questions retrieve nothing useful.

**Advanced RAG (what every senior should know):**

1. **Chunking.** Fixed-size (512 tokens, 50 overlap) is baseline. Semantic chunking groups sentences by embedding similarity. **Parent-document retrieval**: retrieve precise small chunks, pass the parent paragraph/section. **Sentence-window** (LlamaIndex): retrieve a sentence, pass it ± N sentences.

2. **Hybrid search (BM25 + vector) with Reciprocal Rank Fusion.** Per Anthropic's Sept 2024 *Introducing Contextual Retrieval* announcement, contextual embeddings + contextual BM25 reduced the top-20-chunk retrieval failure rate from **5.7% → 2.9% (49% reduction)**; adding reranking pushed it to **1.9% (67% reduction)**. Hybrid is the default in production for any RAG over content with IDs, proper nouns, version numbers.

3. **Re-ranking.** Retrieve 100–150 candidates with cheap bi-encoders, rerank top 20 with a **cross-encoder** (Cohere `rerank-3`, Voyage `rerank-2`, open-source MS-MARCO MiniLM). ColBERT-style late interaction is the middle ground.

4. **Contextual retrieval (Anthropic).** Before embedding each chunk, prepend a model-generated context snippet that situates that chunk within the whole document ("This chunk is from the ACME Q3 2024 earnings report and discusses…"). 35% reduction from contextual embeddings alone per Anthropic's published figures.

5. **Query transformations.**
   - **HyDE** — generate hypothetical answer, embed *it*, retrieve.
   - **Multi-query** — N rephrasings, union/dedup.
   - **RAG-Fusion** — multi-query + RRF.
   - **Step-back prompting** — ask a more general version first.
   - **Query decomposition** — split compound questions.

6. **Self-RAG / Corrective RAG (CRAG).** Model decides *whether* to retrieve, evaluates retrieved docs, re-retrieves on low-quality results. Implement as a small LangGraph.

7. **Agentic RAG.** Retrieval is a tool the agent decides to use, not a fixed step. Can re-query, clarify, escalate.

**Decision framework — RAG vs fine-tuning vs prompting.**
| | Use when |
|---|---|
| **Prompting + few-shot** | Behaviour/format change, small scope, knowledge fits in context |
| **RAG** | Knowledge changes (docs, products, internal data); need citations; need fresh info |
| **Fine-tuning (LoRA/QLoRA, DPO/GRPO)** | Consistent style/format unreachable by prompting; domain-specific reasoning; latency/cost (small tuned model beats prompted big model) |
| **RAG + fine-tune** | High-stakes, regulated; want grounding *and* style |

**LoRA/QLoRA/DPO/GRPO in one line each.** LoRA freezes the base, trains low-rank adapter matrices (rank 8–64) on Q/K/V/O projections. **QLoRA** adds 4-bit NF4 quantisation + paged optimisers — lets you tune 70B on a single A100. **DPO** has largely replaced PPO-based RLHF in 2025–2026 because it needs no separate reward model. **GRPO** (DeepSeek-R1) drops the reference model entirely.

## 1.5 Prompt engineering

- **Zero-shot / few-shot**: with strong 2026 models, zero-shot often wins; few-shot still matters for tricky structured outputs.
- **Chain-of-thought**: with reasoning models (GPT-5 thinking, Claude extended thinking), CoT is built-in — adding manual CoT can hurt.
- **ReAct (Reason + Act)**: interleave `Thought → Action → Observation`. Foundational agent pattern.
- **Self-consistency**: N CoT chains, majority vote. Mostly superseded by reasoning models.
- **Structured outputs:**
  - **OpenAI Structured Outputs** with JSON Schema → guaranteed schema conformance.
  - **Anthropic structured outputs** — per Anthropic API release notes, GA on Claude Sonnet 4.5+/Opus 4.5+/Haiku 4.5+ ("Structured outputs are now generally available on the Claude API… GA includes expanded schema support, improved grammar compilation latency, and a simplified integration path with no beta header required"). The `output_format` parameter moved to `output_config.format`.
  - **Pydantic / Zod**: validate after parsing.
- **XML tags vs JSON in prompts**: Anthropic models are explicitly trained to attend to XML tags — `<context>...</context>`, `<instructions>...</instructions>`. OpenAI is more JSON/markdown-friendly. Match the house style.
- **System vs user prompts**: system = stable identity/policy; user = the task. Put cacheable content (large schema, long docs) at the *top* because prompt caching is prefix-based.

**Prompt caching (production-critical in 2026):**
- **Anthropic** — per the Claude API prompt-caching docs, the minimum cacheable prefix is **4,096 tokens for Claude Mythos Preview / Opus 4.7 / Opus 4.6 / Opus 4.5** and **1,024 tokens for Sonnet 4.6 / Sonnet 4.5 / Opus 4.1**. Default 5-min TTL; **1-hour TTL** is available on Opus 4.5+ / Sonnet 4.5 / Haiku 4.5. **As of Feb 5, 2026, prompt caching uses workspace-level isolation** on the Claude API, Claude Platform on AWS, and Microsoft Foundry. Anthropic publicly reports cache reads "cost significantly less than uncached input tokens," with industry-cited 90% read discounts.
- **OpenAI**: automatic on GPT-5 family, 50% discount on cached input tokens.

## 1.6 Function calling / tool use — under the hood

Pass tool definitions (name, description, JSON schema for parameters) alongside messages. Model either emits text *or* a structured tool-call message (`tool_use` block for Anthropic, `tool_calls[]` for OpenAI). Your client executes the tool, returns the result in a `tool_result` / `tool` message, and the model continues.

**Senior details:**
- **Parallel tool calls** (OpenAI, Anthropic, Gemini): multiple tool calls in one assistant turn, execute concurrently, return all.
- **Tool choice**: `auto` (default), `required`, `{type: "function", function: {name: "x"}}` (force a specific tool).
- **Strict mode** (both vendors): strict JSON-schema enforcement; eliminates "missing field" bugs.
- **Schema design rules.** Names: `lowercase_snake`. Descriptions: action-oriented; include *when not to call*. Parameters: tight schemas (enums, regex, min/max). Fewer than ~30 tools per agent — accuracy degrades past that.

## 1.7 Agents

- **ReAct** — baseline loop `LLM → tool_call → tool_result → LLM → ...`. `create_react_agent` (LangGraph) / `create_agent` (LangChain 1.0) are prebuilt.
- **Planning** — planner LLM produces a plan, executor follows. For long-horizon or HITL-gated work.
- **Reflection** — critic LLM evaluates and asks for revision.
- **Multi-agent architectures** (per LangChain's multi-agent guide):
  - **Supervisor** — one orchestrator routes to specialists. **Dominant production pattern in 2026.**
  - **Swarm / handoff** — agents pass control directly (support chatbots: billing → tech → sales).
  - **Hierarchical** — supervisor-of-supervisors; scales past ~8–10 workers.
  - **Network** — any-to-any; hardest to debug.
- **When NOT to use an agent.** Single-step retrieval, deterministic workflows, strict latency SLAs. Per CallSphere's published May 2026 supervisor-pattern eval, the supervisor pattern is roughly **3× the cost** of a single mega-agent for an **~18-point lift in success rate** (on a 200-task heterogeneous suite, gpt-4o-2024-08-06). Stay single if a one-agent system clears ~85% on your eval.

## 1.8 Context engineering

- **Buffer memory** — last N messages verbatim.
- **Summary memory** — periodic summarisation.
- **Vector-backed memory** — embed past turns, retrieve relevant ones.
- **Server-side compaction** — per Anthropic's *Context windows* docs, **"available in beta for Claude Mythos Preview, Claude Opus 4.7, Claude Opus 4.6, and Claude Sonnet 4.6"**, and "the primary strategy for context management" for long-running agentic workflows.
- **Context awareness** — per the same Anthropic docs, **"Claude Sonnet 4.6, Claude Sonnet 4.5, and Claude Haiku 4.5 feature context awareness. This capability lets these models track their remaining context window (that is, 'token budget') throughout a conversation."**
- **1M context windows** — per Anthropic's March 2026 announcement, Opus 4.6 and Sonnet 4.6 support **1M tokens GA at standard pricing**, with up to 600 images/PDF pages per request. Anthropic's own benchmarks for Opus 4.6 cite **needle-in-haystack going from 91.9% → 78.3% across the full 1M window** — lost-in-the-middle is real. Mitigations: put critical info at start *and* end; multi-needle prompts ("list the 5 most relevant sections first"); set extended-thinking effort to high on >500k inputs.

## 1.10 Cost & latency optimisation

- **Model routing** — biggest lever. Per the Lifetideshub LangGraph guide, **using mini models on supervisors and reflectors while keeping full Sonnet 4.6 / GPT-5 on workers "cuts costs 60–70% with zero impact on routing reliability."**
- **Prompt caching** — cache the schema, system prompt, few-shot examples.
- **Semantic cache** — hash semantically-similar queries (GPTCache, Redis Vector).
- **Streaming** — emit tokens via SSE/WebSocket so TTFT is sub-second.
- **Speculative decoding** — small draft model proposes tokens, big model verifies in parallel. 2–3× speedup with no quality loss; enabled server-side on most providers.
- **Batching** — OpenAI Batches API + Anthropic Message Batches API → 50% discount for non-realtime. Per Anthropic's release notes, **max_tokens cap raised to 300k on Claude Opus 4.6 and Sonnet 4.6 with the `output-300k-2026-03-24` beta header**.
- **Prompt compression** — LLMLingua, RECOMP. Use cautiously.

## 1.11 Safety & guardrails

- **Input validation** — regex/schema before the LLM sees user input.
- **Prompt-injection defence** — separate trusted instructions (system) from untrusted data (user content, retrieved docs, tool outputs); never let retrieved content override system rules.
- **Output validation** — schema-validate every LLM output; for SQL, parse with `sqlglot` and statically reject forbidden statements.
- **PII detection** — Presidio (Microsoft), LLM-Guard, AWS Comprehend PII.
- **Production stack** (per the Spheron 2026 NeMo Guardrails on GPU Cloud guide):
  - **Llama Prompt Guard 2 (86M)** — 20–50ms first-pass injection detector.
  - **LlamaGuard 3 (8B)** — classifies inputs and outputs into hazard categories.
  - **NeMo Guardrails (NVIDIA)** — Colang DSL for programmable rails. Per the Towards Data Science benchmark (*NeMo Guardrails, the Ultimate Open-Source LLM Security Toolkit*), NeMo blocked **16/18 (89%)** of an 18-question prompt-injection set; Llama Guard scored **67%** on the same set by its own category-detection criterion (both happened to miss the same two questions). Use both layered — they fail differently in production.
  - **Guardrails AI** — Python validators for PII, toxicity, profanity, topics.

# SECTION 2 — MCP Deep Dive

## 2.1 Why MCP exists

Per Anthropic's original November 2024 announcement, the problem was the **M×N integration problem**: every AI app needed custom code for every tool. 3 models × 10 tools = 30 bespoke integrations. MCP turns this into M+N — build one MCP server, plug into any MCP-compatible host. Per the 2025-11-25 modelcontextprotocol.io spec, MCP "standardises how to integrate additional context and tools into the ecosystem of AI applications."

**On December 9, 2025, the Linux Foundation announced the formation of the Agentic AI Foundation (AAIF)**, to which Anthropic donated MCP — making it a vendor-neutral standard (per the Linux Foundation press release dated Dec 9, 2025, San Francisco). Per Gartner's *2025 Software Engineering Survey* cited by k2view, **"by 2026, 75% of API gateway vendors and 50% of iPaaS vendors will have MCP features."** Per the official MCP Foundation blog (Dec 9, 2025), the registry passed **"more than 10,000 published MCP servers"** at AAIF formation.

## 2.2 Architecture

```
┌─────────────────────────────┐
│  HOST (Claude Desktop /     │
│  Cursor / your app)         │
│   ┌───────┐ ┌───────┐       │
│   │Client │ │Client │ ...   │
│   └───┬───┘ └───┬───┘       │
└───────┼─────────┼───────────┘
        │JSON-RPC │JSON-RPC
        │ stdio   │ Streamable
        │         │ HTTP
   ┌────▼────┐ ┌──▼─────────┐
   │ Server  │ │  Server    │
   │ (local) │ │ (remote)   │
   └─────────┘ └────────────┘
```

- **Host** — the AI app (Claude Desktop, Claude Code, Cursor, your Next.js chat).
- **Client** — one per server, manages a JSON-RPC session.
- **Server** — a focused service exposing Tools / Resources / Prompts.

## 2.3 The three primitives

- **Tools** — model-controlled functions. Typed via JSON Schema. **This is what your text-to-SQL server uses.**
- **Resources** — app-controlled read-only data, addressed by URI. The *application* decides when to inject. Per Anthropic guidance, "for datasets that exceed the model's context window or return responses larger than roughly 50 KB, use resource URIs instead of stuffing data into tool responses."
- **Prompts** — server-defined reusable templates the user can invoke (slash-commands).

## 2.4 Advanced primitives (2026 senior depth)

- **Sampling** — method `sampling/createMessage` (server → client). Per the 2025-11-25 spec, this "enables servers to leverage AI capabilities — with no server API keys necessary." Inverts the usual control flow. The 2025-11-25 spec added `tools` and `toolChoice` (`auto|required|none`) inside sampling (SEP-1577), enabling server-driven multi-turn agent loops on the client's LLM. Human-in-the-loop is mandated: "there SHOULD always be a human in the loop with the ability to deny sampling requests." Minimal example:
```json
{"jsonrpc":"2.0","id":1,"method":"sampling/createMessage",
 "params":{"messages":[{"role":"user","content":{"type":"text","text":"..."}}],
   "modelPreferences":{"hints":[{"name":"claude-3-sonnet"}],"intelligencePriority":0.8,"speedPriority":0.5},
   "systemPrompt":"You are a helpful assistant.","maxTokens":100}}
```
- **Roots** — method `roots/list`. Per the 2025-11-25 spec, **"Roots define the boundaries of where servers can operate within the filesystem."** Each root is `{uri: "file:///...", name: "..."}`. Notification on change: `notifications/roots/list_changed`. Solves scoping for multi-repo workspaces.
- **Elicitation** — introduced in the 2025-06-18 spec; **URL mode added 2025-11-25 (SEP-1036)**. Method `elicitation/create`. Two modes: `form` (JSON-Schema-validated, primitives only, **MUST NOT** be used for secrets) and `url` (out-of-band for OAuth/API-keys). Responses: `accept | decline | cancel`. URL mode emits `notifications/elicitation/complete`; `URLElicitationRequiredError` is code **-32042**.
- **Tasks (SEP-1686, experimental)** — per modelcontextprotocol.io: **"Tasks were introduced in version 2025-11-25 of the MCP specification and are currently considered experimental."** Methods: `tasks/get`, `tasks/result`, `tasks/list`, `tasks/cancel`; notification `notifications/tasks/status`. Tools advertise `execution.taskSupport: "required" | "optional" | "forbidden"`. Status lifecycle: `working → {input_required ↔ working} → completed | failed | cancelled`. For long-running async ops (20-minute ETL, data pipelines).

## 2.5 Transports

- **stdio** — local, child process, JSON-RPC over stdin/stdout. Default for desktop.
- **HTTP + SSE** (legacy, 2024-11-05 spec): two endpoints. **Being retired.**
- **Streamable HTTP** (current, **introduced 2025-03-26**) — per the 2025-11-25 transports doc, **"This replaces the HTTP+SSE transport from protocol version 2024-11-05."** Single MCP endpoint that handles POST and GET. Server replies with either `application/json` or `text/event-stream`. Supports session via `MCP-Session-Id`, resumable streams via `Last-Event-ID`, version header `MCP-Protocol-Version: 2025-11-25`. Works behind serverless / Lambda.

## 2.6 The underlying protocol: JSON-RPC 2.0

Every MCP message is a JSON-RPC 2.0 envelope (`jsonrpc`, `method`, `params`, `id` or notification, `result`/`error`). Stateful within a session.

## 2.7 Capability negotiation / initialization handshake

Three phases per the 2025-11-25 lifecycle spec: **Initialization → Operation → Shutdown.**

1. Client → server: `initialize` with `protocolVersion`, `capabilities`, `clientInfo`.
2. Server → client: response with its `protocolVersion`, `capabilities`, `serverInfo`, optional `instructions`.
3. Client → server: `notifications/initialized` before any other non-ping traffic.

Client capabilities: `roots`, `sampling`, `elicitation`, `tasks`, `experimental`. Server capabilities: `prompts`, `resources`, `tools`, `logging`, `completions`, `tasks`, `experimental`. Sub-caps: `listChanged`, `subscribe`. Version negotiation echoes if compatible, else downgrades.

## 2.8 Authentication — OAuth 2.1 (June 2025 spec, refined Nov 2025)

Per the 2025-11-25 authorization spec: **"A protected MCP server acts as an OAuth 2.1 resource server… An MCP client acts as an OAuth 2.1 client."**

Required standards:
- OAuth 2.1 draft-13
- RFC 8414 (Authorization Server Metadata)
- RFC 7591 (Dynamic Client Registration)
- **RFC 9728 — Protected Resource Metadata (required)**
- **RFC 8707 — Resource Indicators (required `resource=` parameter on token requests)**
- OAuth Client ID Metadata Documents (added 2025-11-25)

Discovery: server **MUST** publish PRM at `.well-known/oauth-protected-resource` and/or return `WWW-Authenticate: Bearer resource_metadata="..."` on a 401. PRM contains `authorization_servers`. AS discovery order: `/.well-known/oauth-authorization-server` → `/.well-known/openid-configuration` (**OIDC Discovery 1.0 support added 2025-11-25**, PR #797). PKCE S256 mandatory. **Token passthrough is forbidden.**

## 2.9 MCP vs function calling vs REST

- **Function calling** is vendor-specific (OpenAI tools, Anthropic tool use, Gemini function calling). **MCP is a protocol on top that abstracts them.** An MCP server works with any MCP-compatible client regardless of LLM.
- Function calling happens *inside one conversation*. MCP is the plumbing that **makes the tools discoverable in the first place** — `tools/list` returns a dynamic, model-readable manifest.
- REST is transport. MCP adds: standardised discovery, capability negotiation, three primitive types, prompt templates as first-class, sampling, structured auth, structured streaming.
- **Biggest win — reusability**: build the GitHub MCP server once, plug into Claude Desktop, Cursor, your Next.js chat, Claude Code without rewriting integrations.

## 2.10 Why enterprises build internal MCP servers

1. **One integration layer for many AI surfaces.**
2. **Vendor flexibility.**
3. **Governance and audit** — every call is JSON-RPC, fully serialisable.
4. **Capability discovery** — new tools land in all clients on next `tools/list`.
5. **Separation of concerns** — data team owns the server, product teams consume.

## 2.11 Pitfalls & best practices

- **Tool poisoning.** Per Invariant Labs' April 2025 disclosure *MCP Security Notification: Tool Poisoning Attacks*, malicious instructions hidden in tool `description` (or annotations) get read as trusted context. The 2025-11-25 spec acknowledges: "descriptions of tool behavior such as annotations should be considered untrusted, unless obtained from a trusted server." Variants: **direct poisoning**, **tool shadowing** (a malicious server rewrites how the agent calls another trusted server's tool), **rug-pull** (clean on install, mutates later — Invariant's `whatsapp-takeover.py` demo leaked WhatsApp chat history), **output poisoning** (payload in tool results, not descriptions). Real disclosed incidents:
  - **CVE-2025-6514** in `mcp-remote` — OAuth command injection, disclosed by **JFrog Security Research on July 9, 2025**, CVSS **9.6**. Affected ~437K downloads per Docker/vulnerablemcp.info (some sources cite higher figures up to ~559K).
  - **CVE-2025-54136 "MCPoison"** — persistent code execution in Cursor IDE, Aug 2025.
  - **postmark-mcp npm backdoor** (Sept 2025) — first in-the-wild malicious MCP server.
  - **CVE-2025-68143/68144/68145** — Anthropic `mcp-server-git` RCE chain (early 2026).
  - Mitigations: pin tool versions/hashes, run `invariantlabs-ai/mcp-scan`, isolate dataflow between servers.
- **Tool description bloat** — per the Apigene 2026 *LangChain vs MCP* analysis, one team "measured 72% of the context window consumed by tool definitions in a multi-server setup." Solutions: lazy-load tools, route a query to one relevant server, use an MCP gateway.
- **stdio in web servers** — don't. The official `langchain-mcp-adapters` README explicitly warns: **"MCP's stdio transport was designed primarily to support applications running on a user's machine. Before using stdio in a web server context, evaluate whether there's a more appropriate solution. For example, do you actually need MCP? or can you get away with a simple `@tool`?"** Use Streamable HTTP for remote.
- **Versioning** — embed a version in capability response; bump on breaking changes.

## 2.12 How a DB-querying MCP server is built

This is your specific project. Architecture:
```
LLM (host) ── MCP client ──[Streamable HTTP / stdio]──► MCP Server
                                                            │
                                              Tools:               Resources:
                                              - list_tables        - schema://full
                                              - get_schema         - sample_data://...
                                              - execute_query
                                              - explain_plan
                                                  │
                                              ┌───┴───┐
                                              │  DB   │ (read-only role)
                                              └───────┘
```

Typical tool names (matching real MCP servers — Microsoft's `mssqlclient-mcp-server`, Anthropic's Postgres reference server, Azure MCP Server's Postgres tools, the 851-labs `db-explorer-mcp`):

- `list_tables` — table names + row counts.
- `describe_table` / `get_table_schema` — columns, types, FKs, indexes.
- `get_schema` — full schema (DDL or JSON).
- `execute_query` / `query` — read-only SELECT with row cap.
- `get_query_plan` — `EXPLAIN ANALYZE`.

**LLM's loop:** receive question → `list_tables` → `describe_table` on candidates → draft SQL → optionally `validate_sql` → `execute_query` → summarise.

**Senior insight.** Microsoft's own SQL MCP Server (per learn.microsoft.com) **deliberately does NOT support NL2SQL** in its core design — it exposes Data API builder's deterministic query builder instead, because "models aren't deterministic, and complex queries are the most likely to produce subtle errors." Knowing this option is a senior signal.

**Q&A pair — "Why MCP and not just a REST API?"** — "Vendor portability and discovery. The same server serves Claude Desktop, Cursor, Claude Code, and our custom Next.js chat without per-client wrappers. Tools are introspectable at runtime, so when we add `list_views` next week, every client surfaces it automatically. And the OAuth 2.1 + .well-known discovery from the June 2025 spec means our auth model is standard, not bespoke."

```python
from langchain_core.runnables import RunnablePassthrough, RunnableLambda
```

## 3.2 LangGraph (the stateful runtime — what the CTO probably actually wants)

LangGraph reached **stable v1.0 in October 2025** (per AgentsIndex's April 2026 analysis citing the LangChain v1.0 announcement). It models agent execution as a **directed graph of nodes** sharing a typed `State` object, with `add_node`, `add_edge`, `add_conditional_edges`. Every transition is persisted through a **checkpointer**.

**v1.x API:**
```python
from langgraph.graph import StateGraph, MessagesState, START, END
from langgraph.checkpoint.postgres import PostgresSaver
from langgraph.prebuilt import ToolNode, tools_condition
from langchain.agents import create_agent  # 1.0 entrypoint

builder = StateGraph(MessagesState)
builder.add_node("call_model", call_model)
builder.add_node("tools", ToolNode(tools))
builder.add_edge(START, "call_model")
builder.add_conditional_edges("call_model", tools_condition)
builder.add_edge("tools", "call_model")

graph = builder.compile(checkpointer=PostgresSaver(...))
```

**Checkpointing & persistence.** Per the LangChain blog *Making it easier to build human-in-the-loop agents with interrupt*: **"Every step of the graph, it reads from and then writes to a checkpoint of that graph state… This makes it possible to pause execution of the graph half way through, and then resume after some time."** Backends: `InMemorySaver`, `SqliteSaver`, `PostgresSaver`, `AsyncPostgresSaver`. Each thread keyed by `thread_id`.

**HITL with `interrupt()`** (per LangChain HITL middleware docs):
```python
from langchain.agents import create_agent
from langchain.agents.middleware import HumanInTheLoopMiddleware
from langgraph.checkpoint.memory import InMemorySaver

agent = create_agent(
    model="claude-sonnet-4-6",
    tools=[execute_sql, read_data],
    middleware=[HumanInTheLoopMiddleware(interrupt_on={
        "execute_sql": {"allowed_decisions": ["approve", "reject"]},
        "read_data": False,
    })],
    checkpointer=InMemorySaver(),
)
# Resume after human decision:
from langgraph.types import Command
agent.invoke(Command(resume={"decisions": [{"type": "approve"}]}), config)
```
Decision types: `approve`, `edit`, `reject`, `respond`.

**Time-travel debugging** — every state is checkpointed, so you can rewind to any checkpoint, edit state, re-run. Essential for "why did the agent do X?" debugging.

**Production deployments** — per LangChain's official Oct 22, 2025 v1.0 announcement (langchain.com/blog/langchain-langgraph-1dot0), LangGraph powers **"production deployments at companies including Uber, JP Morgan, BlackRock, Cisco, LinkedIn, and Klarna."**

**Prebuilt agents.**
- `create_agent(model, tools, ...)` — canonical LangChain 1.0 ReAct agent with middleware (HITL, summarisation, tool selection).
- `create_react_agent(...)` — LangGraph-prebuilt graph equivalent.
- **`langgraph-supervisor`** — per the official reference: "We now recommend using the supervisor pattern directly via tools rather than this library for most use cases. The tool-calling approach gives you more control over context engineering."

**Multi-agent supervisor (code).**
```python
from langgraph.prebuilt import create_react_agent
from langgraph_supervisor import create_supervisor
from langchain_openai import ChatOpenAI

sql_writer = create_react_agent("openai:gpt-5", tools=[get_schema, draft_sql], name="sql_writer")
sql_validator = create_react_agent("openai:gpt-5-mini", tools=[validate_sql], name="sql_validator")
result_summarizer = create_react_agent("openai:gpt-5-mini", tools=[execute_sql], name="summarizer")

workflow = create_supervisor(
    [sql_writer, sql_validator, result_summarizer],
    model=ChatOpenAI(model="gpt-5"),
)
app = workflow.compile(checkpointer=PostgresSaver(...))
```

## 4.5 Text-to-SQL deep dive

**Schema-aware prompting** is non-negotiable. Per the Medium *Bridging Natural Language and Databases* guide, the number-one failure mode is "hallucinated schemas and SQL" — the LLM invents `Sales` tables or `customer_name` columns that don't exist.

**Schema RAG (production pattern for large DBs).** For DBs with hundreds of tables, embed table + column descriptions in a vector index, retrieve top-k by semantic similarity, pass only the relevant subset to the LLM. Per the Medium guide: *"you embed table and column descriptions into a vector index, and when a user asks a question, you search for the most relevant tables/columns by semantic similarity… This retrieves a subset of schema (say the top 5 tables and top 20 columns related to the query)."* Pair with explicit "Use ONLY the column names provided" instructions.

**TAG (Table-Augmented Generation).** Per the Promethium 2026 guide, also retrieve sample rows so the LLM sees real value formats (date format, enum values, NULL conventions).

**Few-shot examples** of (question, SQL) pairs grounded in *your* schema. Index past validated queries; retrieve similar ones at query time.

**Query validation pipeline:**
1. **Parse** — `sqlglot.parse(...)`.
2. **Static checks** — reject `DROP|DELETE|UPDATE|INSERT|TRUNCATE|GRANT|REVOKE|ALTER`. Cap `LIMIT`. Reject queries without a `WHERE` on huge tables.
3. **Plan check** — `EXPLAIN`; reject if estimated cost > threshold.
4. **Execution sandbox** — read-only DB role + statement-timeout (e.g., 10s) + row-limit (e.g., 1000).
5. **LLM judge** (optional) — "does this SQL answer the question?"

**Self-correction loop.** Per Cesar Schneider's *Implementing an Effective MCP Tool for Text-to-SQL* (2026) and the DIN-SQL paper (which achieved **85.3% execution accuracy on the Spider test set with GPT-4 + decomposition + self-correction**, and 78.2% with Codex Davinci): on DB error, feed the error back to the LLM and retry up to N times. Only retry on **error**, not on empty results (empty might be correct).

> **2026 SOTA reference numbers** (drop these in interview): CHASE-SQL (Google, ICLR 2025) achieves **73.0% on the BIRD test set** and **87.6% on the Spider test set** via multi-path reasoning and preference-based selection. MCI-SQL (VLDB 2026) reports **74.45% on BIRD dev / 76.41% on BIRD test**. These are research-grade numbers on academic schemas — production single-schema systems with strong schema-RAG + few-shot can easily exceed 90% execution accuracy.

**Handling ambiguity.** Build a clarification node. If the question is under-specified ("show me sales"), the agent asks "for which time period and region?" before generating SQL. Use the MCP **elicitation** primitive if surfaced, or LangGraph `interrupt()` otherwise.

**Multi-turn ("show me users from last query but filter by Romania").** Store previous SQL + result schema in `MessagesState` or a custom `TypedDict`. On follow-up, planner sees previous turn's SQL and either appends a `WHERE` or re-plans. **Don't pass full result rows back into context** — keep a summary + the SQL.

**Multi-agent design for text-to-SQL** (Section 4.4 Option C expanded):
- **Planner agent** — decompose complex questions into atomic SQL-able sub-questions.
- **SQL writer agent** — schema-aware, few-shot, structured-output (`{sql: ..., reasoning: ...}`).
- **SQL validator agent** — static parse + rule-based + optional LLM critic.
- **Executor** — calls MCP `execute_query`. HITL on heavy queries.
- **Result summariser agent** — turn rows into NL answer with citations to the SQL.

## 4.6 Preventing destructive queries

Layered defence:
1. **DB-level**: dedicated **read-only role**, no DDL/DML grants. **Primary control.**
2. **Network**: VPC, egress only to DB read replica.
3. **SQL parsing**: `sqlglot` AST inspection — reject anything other than `SELECT` and CTEs.
4. **Prompt level**: system prompt forbids destructive verbs.
5. **Tool level**: MCP `execute_query`'s schema documents "SELECT only," server-side validation.
6. **HITL on heavy queries**: cost estimate > threshold → human approval.

> **Trade-off note.** "Prompt-level guardrails are defence-in-depth, not your primary control. **The DB role is your primary control.** A prompt-injection can override the system prompt; it cannot override Postgres RBAC."

## 4.7 Evaluation strategy

**Offline (regression suite, runs on every PR):**
- 50–200 (question, expected SQL, expected result set) examples covering major tables.
- **Metrics:**
  - **Execution Accuracy (EX)** — run predicted SQL, compare result set to gold. The standard text-to-SQL metric per Spider/BIRD.
  - **Exact Match** — string equality (too strict; secondary signal).
  - **Semantic similarity** — embed predicted vs gold SQL, cosine.
  - **Ragas SQL Query Equivalence** — built-in Ragas metric.
  - **Valid Efficiency Score (VES)** — BIRD's metric combining correctness and efficiency.
  - **Tool-call accuracy** — did the agent call the right MCP tools in the right order?
- **Tooling**: LangSmith `evaluate(...)` with a custom executor running SQL against a snapshot DB.

**Online:**
- **Implicit**: did the user re-ask, accept the answer (thumbs/copy/export).
- **Sampled LLM-as-judge** on production traces: faithfulness, answer relevance.
- **Hard failure metrics**: SQL exec error rate, tool-loop count, p50/p95/p99 latency, token cost per query.

## 4.8 Observability

**LangSmith should capture:**
- Full graph state at each node.

## 4.9 Security

- **SQL injection via prompt injection** — covered (DB role + parse + static checks).

# SECTION 5 — System Design Q&A

### Q: "Design a chatbot that queries our internal databases."

**Strong skeleton.**
1. Clarify: read-only or read-write? How many tables? Roughly how many users? Latency SLO? Compliance (PII, audit)?
2. High-level: Next.js UI (SSE) → API gateway (auth, rate-limit) → LangGraph orchestrator (Postgres checkpointer) → MCP server (read-only DB role) → DB read replica. LangSmith. NeMo Guardrails. Prompt-cached schema.
3. Walk through the graph: planner → schema-RAG → SQL writer → validator → MCP execute_query → summariser → reflection.
4. Eval: offline regression in LangSmith (execution accuracy), online sampled LLM-judge.
5. Trade-offs: vector DB (pgvector if Postgres), model (Sonnet 4.6 for cost/quality, GPT-5-mini for router).

### Q: "How would you scale this to 10k concurrent users?"

- **Stateless agent layer**; horizontally scalable behind LB; `thread_id` carries state.
- **Checkpointer**: `AsyncPostgresSaver` + PgBouncer. Move long-lived state to Redis if Postgres bottlenecks.
- **MCP transport**: Streamable HTTP (stateless on the wire) over stdio (single-process). Per the MCP 2026 roadmap published on blog.modelcontextprotocol.io: **"A stateless HTTP transport variant is in review. This means MCP servers can scale horizontally behind standard load balancers without maintaining persistent SSE connections — critical for high-throughput microservices."**
- **DB**: read replicas; statement-timeout; per-query row caps. Semantic cache for hot queries.
- **LLM**: provider rate limits are the real ceiling. Negotiate org-level RPM quota; spread across providers (Sonnet 4.6 primary, GPT-5 secondary via routing); Batches API for non-interactive.
- **Streaming**: SSE behind CDN with sticky sessions; WebSockets if bidirectional.
- **Cost**: at 10k concurrent users, prompt caching savings dwarf every other optimisation.

### Q: "How do you handle multi-tenancy where each user has different DB access?"

- **Pass identity to the agent**: user JWT → agent context → MCP client headers → MCP server validates and resolves to per-tenant DB role.
- **Per-user/role pools** server-side; never share connections across tenants.
- **Row-level security** in Postgres (`SET app.current_user = ...` + RLS policies).
- **Audit**: every query logged with `tenant_id`.
- **Schema-RAG scoping**: per-tenant index if schemas diverge; otherwise filter retrieved tables by tenant permission.

### Q: "Walk me through how you'd add RAG on top of this."

Two RAG layers:
1. **Schema RAG** — vectorise table/column descriptions + sample rows + known-good queries.
2. **Document RAG** for unstructured context (runbooks, wikis, past tickets). Different vector store (or same pgvector with a `kind` column). Planner decides which retrieval to invoke.

### Q: "How do you eval this system before shipping?"

Three layers: (1) Unit-level (LLM-judge on each node in isolation), (2) integration (full graph on a 200-example regression set, EX ≥ X% as release gate), (3) shadow/canary online comparison vs current.

### Q: "How do you prevent the LLM from running destructive queries?"

Section 4.6 layered defence. Lead with **DB-level read-only role** as primary control; everything else defence-in-depth.

### Q: "How would you A/B test two prompts?"

LangSmith experiments: register two prompt versions, route 50/50 by `thread_id` hash, evaluate on same dataset, compare EX + faithfulness + latency + cost. Statistical significance with bootstrap CIs at N≥200.

### Q: "What if the LLM gets stuck in a tool-call loop?"

- Hard cap: `max_iterations` in `create_agent` (default ~10; set to 6 for SQL).
- LangGraph supports a `recursion_limit` on `.invoke(...)`.
- Detect repeated identical tool calls (hash the args), break and ask for clarification.
- Trace in LangSmith with an alert; loops mean ambiguous tool schemas or system prompt.

### Q: "How do you handle long conversations that exceed context window?"

- **Anthropic server-side compaction** on Sonnet 4.6 / Opus 4.6+ (beta).
- **LangGraph summarisation middleware**: token count > 70% of window → summarise oldest N turns.
- **Vector-backed memory**: embed past turns; retrieve only relevant for current query.
- **Externalise large data**: 10k-row result → summarise + store elsewhere; keep pointer + schema in context.
- **1M context** buys time but doesn't eliminate need — Anthropic's Opus 4.6 needle-in-haystack figure of 91.9% → 78.3% across the window still applies.

### Q: "How would you make this work offline / on-premise?"

- **Open-weight model**: Llama 4 70B or DeepSeek V4-Pro on-prem via vLLM, OpenAI-compatible API.
- **Embeddings**: BGE-M3 or Nomic-embed locally.
- **Vector DB**: Qdrant or pgvector on-prem.
- **Guardrails**: LlamaGuard 3 + NeMo Guardrails. Per the Spheron 2026 guide, the production stack is "NeMo Guardrails orchestrating both LlamaGuard 3 8B (for detailed hazard classification) and Llama Prompt Guard 2 86M (as a fast first-pass gate)."
- **Observability**: self-hosted LangSmith Enterprise, or Langfuse (open-source).
- **MCP**: stdio for local; Streamable HTTP behind own reverse proxy with OAuth 2.1 for remote.

# SECTION 8 — Likely Gotcha / Deep-Dive Questions

### "Explain the attention mechanism."
See Section 1.1. Be ready to draw it: token → Q, K, V projections → `softmax(QKᵀ/√d_k)V` → multi-head concat → output projection. Mention causal mask in decoders. Mention KV-cache as the optimisation that makes autoregressive decoding O(n) per token.

### "Encoder vs decoder vs encoder-decoder?"
Encoder = bidirectional, used for embeddings (BERT). Decoder = causal, used for generation (GPT, Claude). Encoder-decoder = both, used historically for seq2seq (T5); largely replaced by decoder-only.

### "Why do LLMs hallucinate?"
Probabilistic next-token prediction has no grounding in truth — optimising for plausible continuation, not correctness. Worse when (a) weak training signal on the domain, (b) under-specified question, (c) asked for specifics the model doesn't have. **Mitigations:** RAG, citations, structured outputs constrained to retrieved entities, faithfulness eval (Ragas), refusal tuning.

### "BPE vs WordPiece vs SentencePiece?"
BPE = byte/char pair merges by frequency (GPT). WordPiece = merges by likelihood gain (BERT). SentencePiece = language-agnostic, trains on raw text including whitespace, supports BPE and Unigram (Llama, Gemini, T5).

### "How does function calling work at the API level?"
Pass `tools=[{name, description, parameters: <JSONSchema>}]`. Model returns text or a `tool_use` block (Anthropic) / `tool_calls` array (OpenAI) with `name` + `input`. You execute, return `tool_result` (Anthropic) / `tool` message (OpenAI). Model continues. Parallel: multiple `tool_use` blocks in one assistant turn.

### "MCP vs OpenAI tools / Anthropic tools / Google function calling?"
- **Vendor function calling** is in-conversation feature, vendor-specific schema. Per Descope's *MCP vs Function Calling*: "OpenAI's function calling, Anthropic's tool use, Google's function calling all have different formats. MCP is a standardised protocol that sits above these."
- **MCP** is a protocol over JSON-RPC exposing tools to *any* MCP client. Adds discovery (`tools/list`), three primitive types, capability negotiation, sampling, elicitation, roots, OAuth 2.1.
- **MCP doesn't replace function calling — it sits above it.** The MCP client converts MCP tools into native function-calling format for whichever LLM is in use.

### "How does LangGraph differ from a state machine?"
- A state machine has finite states and deterministic transitions.
- LangGraph is **a typed, persisted, durable, dynamic graph** where transitions can be conditional on LLM outputs. State is a `TypedDict` updated by nodes. Edges can be conditional functions. Every transition is checkpointed. Supports cycles, parallel execution (`Send` for fan-out), HITL interrupts. **It's a state machine generalised to non-deterministic agent control flow with first-class durability.**

### "What's a checkpointer in LangGraph and why does it matter?"
Pluggable storage backend (`InMemorySaver`, `SqliteSaver`, `PostgresSaver`) persisting graph state at every node transition. Per the LangChain blog: "Every step of the graph, it reads from and then writes to a checkpoint of that graph state." Without it: no HITL, no time-travel, no surviving a kernel restart, no multi-instance horizontal scaling. **In production: `PostgresSaver`. Always.**

### "Async vs stream in LangChain?"
- **Async** — call is non-blocking; you `await` the full final result. `ainvoke`, `abatch`. Doesn't affect TTFT.
- **Stream** — call emits intermediate chunks. `stream` / `astream` for tokens; `astream_events` for fine-grained events (node start/end, tool calls, LLM tokens) — essential for showing intermediate steps in a chat UI.
- They compose: `async for chunk in chain.astream(...)`.

### "How do you debug a misbehaving agent?"
1. **LangSmith traces** — find the request, look at every node's I/O and every LLM call.
2. **Reproduce** with a regression test pinning the exact input.
3. **Bisect** with each node mocked to isolate which one introduced the bad state.
4. **Time-travel** in LangGraph: fork the failing checkpoint, edit state, retry.
5. **Reduce** — simplify the prompt, re-add complexity until it breaks.

# Caveats

- **Model versions move fast.** The Anthropic platform release notes I worked from list Opus 4.7 as current flagship and Sonnet 4.6 as the default chat model, with Sonnet 4.6 GA from February 2026 and Opus 4.6/Sonnet 4.6 1M-token context GA from March 2026. If you sit the interview later than May 2026, re-check the current flagship before walking in.
- **LangChain version churn is real.** This guide is calibrated to LangGraph v1.x (stable since October 2025) and LangChain 1.0's `create_agent`. If you find blog posts using `AgentExecutor`, `create_react_agent` from old `langchain.agents`, or `LCEL` only — those pre-date the modular v1.0 split. Use `langchain-core` + `langgraph` + provider packages.
- **MCP spec is moving.** The 2025-11-25 spec is current at the time of writing (May 22, 2026). The 2026 MCP Roadmap (blog.modelcontextprotocol.io, March 9, 2026 by David Soria Parra) highlights stateless HTTP, Tasks lifecycle iteration, MCP Server Cards, and an enterprise readiness track — those are on the roadmap, **not yet shipped**. Don't claim them as production.
- **Some numbers cited here are vendor-published benchmarks** (Anthropic's contextual retrieval %, Anthropic's needle-in-haystack on Opus 4.6, Morph's LangGraph overhead figure). Treat them as directional, not absolute — and say so in the interview ("per Anthropic's published benchmarks…") rather than presenting them as universal truths.
- **One CVE download count has conflicting reports** — CVE-2025-6514 in `mcp-remote` was disclosed by JFrog Security Research on July 9, 2025 with CVSS 9.6; download counts at time of disclosure range from ~437K (Docker, vulnerablemcp.info) to ~559K (TrueFoundry, Amla Labs). - **The "rebuild MCP chat with LangChain" framing is a hypothesis** based on the interview brief. The real conversation may reveal a totally different motivation (compliance, hiring, vendor exit) — be ready to abandon the wrap-vs-rebuild framing if the real driver is different.
